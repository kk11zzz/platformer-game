# 横板过关游戏迭代大纲（Phaser 3 版）

目录：`/root/.openclaw/游戏/`

目标：做一个类似马里奥的横板过关游戏，核心爽点是「跑、跳、踩怪、过关」。先不追求大而全，先把最小可玩版本做出来，然后一步一步加内容。

技术路线：**Phaser 3 + JavaScript + 浏览器**

选择 Phaser 的原因：

- 社区活跃
- 教程和示例很多
- 文档完整
- 适合 2D 横板过关
- 浏览器可直接运行
- 内置物理、输入、场景、摄像机、精灵、碰撞检测等常用能力
- 对非专业游戏程序员更友好

## 1. 第 1 步：Phaser 游戏初始化

使用：

验收标准：

- 控制台没有错误。

### 1.3 创建第一个 Scene

目标：理解 Phaser 的场景机制。

建议先创建一个 `BootScene` 或 `GameScene`。

## 2. 第 2 步：显示背景和地面

### 2.1 加载或绘制背景

目标：让画面有游戏氛围。

方案：

- 初期先用纯色背景
- 后续替换成背景图
- 后续可以加视差滚动

验收标准：

- 背景填充天空蓝或显示背景图。
- 画面稳定。

### 2.2 创建地面平台

目标：玩家有地方站立。

使用 Phaser 物理：

```js
this.physics.add.staticGroup()
```

验收标准：

- 地面显示在画面底部。
- 地面不会移动。
- 地面可以作为物理碰撞体。

### 2.3 创建悬空平台

目标：为跳跃和关卡设计做准备。

验收标准：

- 至少有一个悬空平台。
- 平台可以被玩家碰撞。
- 平台不会受重力影响。

## 3. 第 3 步：玩家出现并受重力影响

### 3.1 创建玩家 Sprite

目标：玩家出现在场景中。

使用：

```js
this.physics.add.sprite(x, y, 'player')
```

初期没有素材时，可以先用色块/占位图。

验收标准：

- 屏幕上出现玩家。
- 玩家是物理对象。
- 玩家受重力影响。

### 3.2 设置玩家物理参数

目标：让玩家可以正常下落和站立。

参数：

- `setCollideWorldBounds(true)`
- `setGravityY(...)`
- `setSize(...)`
- `setOffset(...)`

验收标准：

- 玩家从空中生成后会落到地面。
- 玩家不会掉出世界。
- 玩家可以站在地面上。

### 3.3 玩家与平台碰撞

目标：玩家不会穿过地面和平台。

使用：

```js
this.physics.add.collider(player, platforms)
```

验收标准：

- 玩家落在地面后停止下落。
- 玩家落在悬空平台上。
- 玩家从下方跳起会撞到平台底部。

---

## 4. 第 4 步：键盘控制玩家移动和跳跃

### 4.1 创建输入对象

目标：监听键盘。

使用：

```js
this.input.keyboard.createCursorKeys()
this.input.keyboard.addKey('A')
this.input.keyboard.addKey('D')
```

按键：

- `A` / `←`：向左
- `D` / `→`：向右
- `W` / `↑` / `空格`：跳跃

验收标准：

- 能检测按键按下和松开。
- 输入状态稳定。
- 不会触发浏览器默认滚动。

### 4.2 左右移动

目标：玩家可以横向移动。

使用：

```js
player.setVelocityX(...)
```

验收标准：

- 按住右键，玩家向右移动。
- 按住左键，玩家向左移动。
- 松手后玩家停止。

### 4.3 跳跃

目标：玩家可以从地面跳起。

使用：

```js
player.setVelocityY(-jumpPower)
```

建议配合：

```js
player.body.blocked.down
player.body.touching.down
```

验收标准：

- 玩家在地面时可以跳。
- 空中不能无限连跳。
- 跳跃高度手感可调。

## 5. 第 5 步：摄像机跟随

### 5.1 创建宽关卡

目标：关卡宽度大于屏幕宽度。

验收标准：

- 地面和平台延伸到屏幕右侧。
- 玩家能向右移动很远。

### 5.2 摄像机跟随玩家

目标：玩家始终保持在可视范围内。

使用：

```js
this.cameras.main.startFollow(player)
this.cameras.main.setBounds(0, 0, levelWidth, levelHeight)
```

验收标准：

- 玩家向右移动时，画面跟随移动。
- 摄像机不会显示关卡外区域。
- 玩家不会轻易跑出可视区域。

### 5.3 背景视差（可选）

目标：让背景移动更自然。

验收标准：

- 背景移动速度慢于前景。
- 画面层次感更强。
- 不影响核心玩法。

验收标准：

验收标准：

### 7.2 敌人与平台碰撞

目标：敌人能站在平台上。

使用：

```js

```

验收标准：

## 8. 第 8 步：玩家与敌人碰撞

这是核心玩法，需要拆得很细。

### 8.1 只做碰撞检测

目标：判断玩家和敌人是否重叠。

使用：

```js
this.physics.add.collider(player, enemies)
```

验收标准：

- 玩家和敌人不会穿过去。
- 碰撞事件能触发。

### 8.2 判断玩家是否从上方踩到敌人

目标：区分「踩怪」和「被怪碰到」。

判定逻辑：

- 玩家上一帧底部在敌人顶部附近
- 玩家当前速度向下
- 玩家与敌人发生重叠
- 满足条件则判定为踩到敌人

可使用：

```js
player.body.touching.down
enemy.body.touching.up
player.body.velocity.y > 0
```

验收标准：

- 玩家从上方落下碰到敌人时，敌人死亡。
- 玩家从侧面碰到敌人时，玩家死亡。

### 8.3 踩怪反馈

目标：踩怪要有爽感。

验收标准：

- 敌人消失或变成压扁状态。
- 玩家被轻微弹起。
- 可以增加简单音效或粒子。

### 8.4 玩家受伤/死亡

目标：碰到敌人侧面会失败。

验收标准：

- 玩家碰到敌人侧面后回到起点。
- 或者进入 `GameOverScene`。
- 死亡后短暂无敌或重新加载当前关。

## 9. 第 9 步：金币/分数系统

### 9.1 创建金币 Group

目标：管理多个金币。

使用：

```js
this.physics.add.group()
```

验收标准：

- 关卡里能放置金币。
- 金币先可以用圆形或矩形表示。
- 金币是物理对象。

### 9.2 玩家收集金币

目标：碰到金币后加分。

使用：

```js
this.physics.add.overlap(player, coins, collectCoin, null, this)
```

验收标准：

- 玩家碰到金币后金币消失。
- 分数增加。
- 已收集金币不会重复计分。

### 9.3 UI 显示分数

目标：玩家能看到当前分数。

使用：

```js
this.add.text(...)
```

验收标准：

- 左上角显示 `Score: 0`。
- 收集金币后数字更新。
- UI 跟随摄像机或固定在屏幕左上角。

## 10. 第 10 步：终点与过关

### 10.1 创建终点区域

目标：玩家到达终点后过关。

终点字段：

```js
{
  x,
  y,
  width,
  height
}
```

验收标准：

- 关卡右侧有终点旗杆/门/区域。
- 玩家碰到终点区域后触发过关。

### 10.2 使用 overlap 检测过关

使用：

```js
this.physics.add.overlap(player, goal, reachGoal, null, this)
```

验收标准：

- 玩家碰到终点后停止游戏循环或进入结算。
- 屏幕显示 `Level Clear` 或类似提示。
- 显示最终分数。

### 10.3 重新开始关卡

目标：玩家可以从头再玩。

验收标准：

- 过关后按任意键或点击按钮重新开始。
- 玩家、敌人、金币、分数重置。

## 11. 第 11 步：关卡数据化

目标：不要硬编码所有物体，而是用数据描述关卡。

### 11.1 设计关卡格式

示例：

```js
export const level1 = {
  width: 3200,
  height: 540,
  spawn: { x: 80, y: 300 },
  platforms: [
    { x: 0, y: 480, width: 900, height: 60 },
    { x: 1000, y: 380, width: 300, height: 40 }
  ],
  enemies: [
    { x: 500, y: 420, patrolLeft: 450, patrolRight: 750 }
  ],
  coins: [
    { x: 300, y: 420 },
    { x: 1100, y: 330 }
  ],
  goal: { x: 3000, y: 380, width: 80, height: 100 }
}
```

验收标准：

- 关卡对象能生成平台、敌人、金币、终点。
- 修改数据即可改变关卡。

### 11.2 多关卡准备

目标：为后续关卡扩展留接口。

验收标准：

- 可以定义 `levels = [level1, level2, level3]`。
- 过关后能加载下一关。
- `GameScene` 能根据关卡编号加载不同数据。

# 最小可玩版本 MVP

MVP 只包含这些：

- 一个 Phaser 游戏场景
- 一个玩家
- 一个地面
- 两个平台
- 一个坑
- 一个巡逻怪兽
- 踩怪死亡
- 被怪碰到死亡
- 三个金币
- 一个终点
- 过关提示
- 死亡重生

MVP 不需要：

- 多关卡
- 复杂动画
- 音效
- 粒子
- 精美素材
- 敌人 AI
- Boss
- 存档
